  <div class="eyebrow">INTELLIGENCE SEARCH SYSTEM · PIPELINE SCHEMATIC</div>
  <h1>Search API <span class="accent-cyan">×</span> Kafka Flow</h1>
  <p class="sub">One request plane that answers fast, one streaming plane that answers well — from raw query to a reranked, cited response.</p>

  <div class="legend">
    <span class="legend-item"><i class="dot cyan"></i>Sync — client / API</span>
    <span class="legend-item"><i class="dot amber"></i>Kafka topic</span>
    <span class="legend-item"><i class="dot violet"></i>Consumer / processor</span>
    <span class="legend-item"><i class="dot slate"></i>Storage / state</span>
  </div>

  <!-- ================= SYNC ENTRY ================= -->
  <div class="plane-label"><span class="tag sync">Sync</span></div>

  <div class="timeline">

    <div class="stage" data-plane="sync">
      <div class="stage-kind">01 · Client → API</div>
      <div class="card">
        <h3><code>POST /api/v1/search/query</code></h3>
        <p>Client sends query, filters, top_k. API mints <code>request_id</code> and <code>correlation_id</code>, validates the payload schema.</p>
      </div>
    </div>

    <div class="stage" data-plane="sync">
      <div class="stage-kind">02 · Search API — Triage</div>
      <div class="card">
        <h3>Query classification</h3>
        <p>Fast in-process check: is this a simple lexical lookup, or does it need multi-strategy retrieval + rerank + synthesis?</p>
        <div class="callout"><b>Fast path —</b> simple lookups skip Kafka entirely: direct BM25 call to Elasticsearch, response returned in the same request/response cycle.</div>
        <p>Complex queries continue below. Job status written to Redis as <code>PENDING</code>, event published to Kafka.</p>
      </div>
    </div>

  </div>

  <!-- ================= ASYNC PIPELINE ================= -->
  <div class="plane-label"><span class="tag async">Async</span></div>

  <div class="timeline">

    <div class="stage" data-plane="topic">
      <div class="stage-kind">03 · Kafka Topic</div>
      <div class="card">
        <div class="topic-name">search.request.raw</div>
        <p>query · filters · request_id · correlation_id (header) · submitted_at</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">04 · Processor</div>
      <div class="card">
        <h3>Query Understanding Processor</h3>
        <p>Rewrites and expands the query (domain acronym normalization — G-SIB, NBFC, LEF), classifies intent, extracts entities and date-range filters.</p>
        <div class="branch fail">on failure → <code>search.request.dlq</code></div>
      </div>
    </div>

    <div class="stage" data-plane="topic">
      <div class="stage-kind">05 · Kafka Topic</div>
      <div class="card">
        <div class="topic-name">search.request.enriched</div>
        <p>normalized query · extracted filters · intent label</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">06 · Processor — fan-out</div>
      <div class="card">
        <h3>Retrieval Processor</h3>
        <p>Runs three strategies in parallel against Elasticsearch: BM25 lexical, dense kNN (mpnet-base-v2 query embedding), and SPLADE sparse.</p>
        <div class="tech-line">top_n per strategy, independently tunable</div>
      </div>
    </div>

    <div class="stage" data-plane="store">
      <div class="stage-kind">Storage</div>
      <div class="card">
        <h3>Elasticsearch</h3>
        <p>dense_vector kNN + BM25 analyzed text + content_type / doc_type filters, in one queried index.</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">07 · Processor</div>
      <div class="card">
        <h3>Fusion (RRF)</h3>
        <p>Combines the three ranked lists with Reciprocal Rank Fusion into one candidate set — no single strategy dominates.</p>
      </div>
    </div>

    <div class="stage" data-plane="topic">
      <div class="stage-kind">08 · Kafka Topic</div>
      <div class="card">
        <div class="topic-name">search.candidates.fused</div>
        <p>fused candidate chunks · RRF scores · strategy provenance</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">09 · Processor</div>
      <div class="card">
        <h3>Reranking Processor</h3>
        <p>Cross-encoder rerank with bge-reranker-large — top ~50 candidates narrowed to the top ~10 most relevant.</p>
        <div class="branch fail">on failure → <code>search.rerank.dlq</code></div>
      </div>
    </div>

    <div class="stage" data-plane="topic">
      <div class="stage-kind">10 · Kafka Topic</div>
      <div class="card">
        <div class="topic-name">search.results.reranked</div>
        <p>top-k chunks · rerank scores · source metadata</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">11 · Processor</div>
      <div class="card">
        <h3>Synthesis Processor</h3>
        <p>Builds a grounded prompt from the reranked chunks and calls the on-prem LLM (GPT-OSS-20B) for a cited answer.</p>
        <div class="branch fail">on failure → <code>search.synthesis.dlq</code></div>
      </div>
    </div>

    <div class="stage" data-plane="topic">
      <div class="stage-kind">12 · Kafka Topic</div>
      <div class="card">
        <div class="topic-name">search.response.final</div>
        <p>synthesized answer · cited chunk ids · confidence / latency breakdown</p>
      </div>
    </div>

    <div class="stage" data-plane="proc">
      <div class="stage-kind">13 · Processor</div>
      <div class="card">
        <h3>Response Writer</h3>
        <p>Persists the final result keyed by <code>request_id</code>, status set to <code>COMPLETED</code>; pushes over WebSocket if the client is subscribed.</p>
        <div class="branch">also emits → <code>search.audit.log</code> for the eval harness (LLM-as-judge, ROUGE-L, citation presence)</div>
      </div>
    </div>

    <div class="stage" data-plane="store">
      <div class="stage-kind">Storage</div>
      <div class="card">
        <h3>Redis / Job Store</h3>
        <p>Holds job status and final payload, keyed by request_id — what the API polls or pushes from.</p>
      </div>
    </div>

  </div>

  <!-- ================= SYNC RETURN ================= -->
  <div class="plane-label"><span class="tag sync">Sync</span></div>

  <div class="timeline">
    <div class="stage" data-plane="sync">
      <div class="stage-kind">14 · API → Client</div>
      <div class="card">
        <h3><code>GET /api/v1/search/results/{request_id}</code> <span style="color:var(--muted); font-weight:400;">or WS push</span></h3>
        <p>Client retrieves the final answer, its cited source chunks, and a per-stage latency breakdown.</p>
      </div>
    </div>
  </div>
